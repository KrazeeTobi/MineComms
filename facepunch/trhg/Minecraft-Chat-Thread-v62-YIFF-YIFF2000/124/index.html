[QUOTE=RivaGe;29153155]I don't get the fuss about mod api. It is supposed to make modding easier but every mod will still have to updated with each MC update. At least it's what I think.[/QUOTE] This is how it works: When "compiled", Java makes class files. It obfuscates them automatically (that means it gives classes random bullshit names) upon "compilation". When someone makes a mod, it "hooks" into a certain class, which is called upon by its obfuscated name. Now, when a new class is added, the obfuscation method changes the names of EVERYTHING, so the old hooks don't work anymore. This is why Bukkit is such a boon to the server modding community: They just have to worry about Bukkit updating to change the hooks so that Bukkits hooks can still be called upon with minimal effort, though with Bukkit being in continuous development this usually means that plugin developers must also make small updates to their mods. Unfortunately, for single player no such development kit exists so people have to rely on others to de-obfuscate shit for them (which could take several days of painful examination of classes, if not weeks) before they can update their mod to work with the new update. Now, with a modding API, the API would *hopefully* maintain a constant list of named hooks that mods could call upon. Upon compilation, the class names would change, but the API would update automatically because *hopefully* the folks at Mojang will have them inter-linked so that nothing changes on the development side while internal workings can change as they always have. So yeah a mod API would be a pretty god-damn big deal.
