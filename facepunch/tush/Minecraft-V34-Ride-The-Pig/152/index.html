Also: [url]http://notch.tumblr.com/post/963183697/working-on-server-side-inventory[/url] :siren: Working on server-side inventory :siren: There are a few stability issues with the client that worry me, and I will try to sort that out before I go on the trip to Finland next week, but more important is getting SMP more playable before then. I&#8217;m currently working on server-side inventory, and it&#8217;s going relatively well. I&#8217;m currently pulling on hard coded strands of spaghetti code, seeing what moves, then refactoring that out into a better designed code. See, when I first wrote the inventory, it was just a test to see if it was fun, and I&#8217;ve stuck to that code ever since since I never really had to change it. I probably would&#8217;ve saved myself some time if I had fixed the code immediately when it was still fresh in my mind. The worst part about refactoring code is that you spend hours working on something only to have it do the exact same thing when you&#8217;re done with it. The best part is that you feel much better about working with that code in the future. It&#8217;s very cleansing, in a boring kind of way. One interesting problem that came up was that of client-side prediction. Since the client is able to accurately predict inventory management 100% (with the exception of crafting, which I&#8217;m making controlled by the server, for modding reasons), there&#8217;s no reason to synch any state other than to tell the server what happens. Or so I thought! For example, another player can throw an item at you while you&#8217;re playing around in your inventory. If you split a stack client side, your inventory might end up being full, but the server thought you had room because of lag, and that item ends up getting lost. So instead of having the server assign new items a specific slot, it just goes &#8220;here, take this if you can&#8221;, and the client says either &#8220;ok, I put it in slot #7&#8221; or &#8220;no&#8221; as a reply. But then what about chests? I can either make chest only operatable by one player at a time, or I can make them fully synched by the server, meaning they would get noticeably laggy. I don&#8217;t want to lock chests to one user, as that&#8217;s prone to griefing, nor do I want to do the fully synchronous inventory management in them as it really ruins the smooth feeling. I could do some clever rollback scheme where the client will predict moves until the server says that a certain move was illegal (like if two players try to take the same item at the same time, only the first player should get it), but that might get annoying. You see yourself picking up an item, and you place it in your inventory, but then it vanishes. [why is tumblr refusing to post some entires?]
