[QUOTE=JWJ;28065317]In light of the crazy ass chunk error loss that occured when Brocraft shit itself, I'm thinking about programming an incremental backup system which will allow you to revert every single file on the server to any point in time in the past 24 hours, by the minute. For example, you could just type in your MC server /revert 2-13-2011 14:28 and have it how it was on the 13th at 2:28pm, or /revert 5 to revert to 5 minutes ago. Every file, from logs to SQLite databases to the world itself will be reverted to make sure no inconsistencies occur. From a technical standpoint the program would do the following: [LIST] [*]If any file is changed, it will record this change and mark the file as an 'active file'. [/LIST] [LIST] [*]In the background, the backup process will start loading the last image of the active file into memory and each changefile for that file since the last image was made. (up to a total of 14 changefiles) [/LIST] [LIST] [*]At the end of every minute, each active file will be locked for writing and loaded into memory. It will be compared with the previous image and a changefile will be generated. This happens very quickly and will only lock one file at a time to ensure the server itself does not lock up waiting on an IO handle. If a file cannot be locked for reading, it will be pushed to the back of the active file queue until it can. If any file cannot be backed up, the entire backup operation for that minute will cancel to ensure no incomplete backups are recorded. [/LIST] [LIST] [*]Every 15 minutes, a full image of a file will be generated by loading up the changefiles from the last 15 minutes and applying them to the last image recorded. [/LIST] [LIST] [*]When a revert is requested, the program will first shut down the Minecraft server. After the minecraft server has terminated, it will go through every file currently backed up, load the last image created prior to the requested time and then apply each changefile up to the requested time. All of the prior operations are done in-memory and then written to the hard drive. Once the revert operation is complete, any changefiles and images past the revert time will be deleted and the server will start itself back up. [/LIST] An incremental backup system is much more efficient than a full backup system because it doesn't require the entire server to stop and it allows for much finer precision in terms of time. In total, every server would be keeping a maximum of 1344 changefiles and 96 images for every file that is changed if the file is changed every minute. This requires a lot of diskspace, but considering every box has 2 terabytes of space I'm not terribly concerned. This should keep me busy, but I think it will be worth it as it isn't something you can get with a non-Brohoster server until someone reimplements my idea. :v:[/QUOTE] This is a great idea and all, except that when the server took a dump I couldn't even connect and it sounds like you want this all to be done within Minecraft itself...
