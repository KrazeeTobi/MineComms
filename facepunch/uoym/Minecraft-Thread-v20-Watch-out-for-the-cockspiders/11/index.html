New blog post. [quote=Notch]First of all, calling blocks &#8220;Tiles&#8221; in the codebase is horribly confusing. Also, TileEntities being named similar to Entities doesn&#8217;t help at all. But the worst piece of design is older than that, dating all the way back to the cave game test build. It&#8217;s a walking null pointer exception waiting to happen. I keep a static Tile[256] list that maps a byte value to a Tile instance, which is used to do all the block logic. For example, to find out what material a block is (say, id 40) I&#8217;d go Tile.tiles[40].getMaterial(). Works fine. But for some reason, I decided to make air have a null block, and be identified instead by its id, 0. This means that everywhere I want to do something with a Tile instance, I have to do something like &#8220;int id = level.getTileAt(x, y, z); if (id==0) return false; else return Tile.tiles[id].getMaterial()==Material.rock;&#8221; when checking for rock instead of just a simple &#8220;return level.getTileAt(x, y, z).getMaterial()==Material.rock;&#8221; A couple of months ago, I made the effort to try to fix this, mainly by removing the ability to even ASK for a raw block type byte from the level, but it was really surprising how many places relied on those bytes. The renderer and level generators, primarely, act on the raw bytes a lot for speed. And additionally, there are a lot of places that rely on a Tile.tiles[getTileAt(x, y, z)]==null to check for air spaces. The problem is just big and scary enough for my laziness to prevent me from fixing it, and it&#8217;s not dangerous enough to force me to fix it (either the game works, or it doesn&#8217;t. It can&#8217;t corrupt saves or anything like that). And the Seecret Friday Update is either coming within a few hours, or some time after many more, due to me attending a birthday party! (Happy birthday, if you&#8217;re reading this. :D)[/quote] [url]http://notch.tumblr.com/post/760983716/my-worst-code-design-decision[/url]
